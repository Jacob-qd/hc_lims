## P0-01 电子签名(国密SM2/SM3)

### 对标基准
StarLIMS: 21 CFR Part 11 全认证 — 电子签名从底层设计，支持多因子认证、密码策略、签名含义声明、审计链。
LabWare: 同 StarLIMS，符合 FDA 21 CFR Part 11 和 EU GMP Annex 11。

### HC-LIMS 当前状态
❌ 基础密码验证 — SignaturePad 组件支持签名绘制，但无国密算法、无签名含义声明、无时间戳、无审计链。

### 功能说明

**核心功能点:**
1. **SM2 签名算法**: 采用国密 SM2 椭圆曲线公钥密码算法，替代 RSA/ECDSA
   - 签名数据: 用户身份 + 签名含义 + 操作内容摘要 + 时间戳
   - 签名存储: 存储完整签名数据包 (签名值、证书链、时间戳)，支持验真
2. **SM3 摘要算法**: 采用国密 SM3 密码杂凑算法，替代 SHA-256
   - 用于文档摘要、原文验证、签名防篡改
3. **数字证书管理**:
   - 支持导入 SM2 国密证书 (支持 GM/T 0009 标准)
   - 证书吊销列表 (CRL) 检查
   - 支持对接 CA 机构 (天威诚信、CFCA、国富安)
4. **签名含义声明** (符合 21 CFR Part 11):
   - 签名时要求用户选择签名含义: "审核通过" "批准" "复核" "编制"
   - 显示签名前最后确认内容摘要
   - 记录签名意图 + 签署时间 + 签署人身份
5. **签名审计链**:
   - 每次签名生成不可篡改的审计记录
   - 包含: 签名人、时间、含义、文档指纹(SM3摘要)、签名值
   - 审计记录不可删除，仅可追加
6. **多方签名工作流**:
   - 支持编制→审核→批准 多级签名
   - 每一步签名后文档状态锁定
   - 拒绝签名需填写理由并退回上一步
7. **防篡改验证**:
   - 报告展示时实时验证签名有效性
   - 检测到篡改后显示红色警告 "签名已失效"
   - 提供签名验真二维码 (其他人扫码可验)

### 技术方案

```
前端                              后端/服务
┌────────────────────┐           ┌─────────────────────┐
│  SignaturePad 升级   │           │  Sign Service        │
│  · 签名绘制界面       │           │  · SM2签名            │
│  · 签名含义选择       │  REST    │  · SM3摘要            │
│  · 密码二次确认       │◄────────►│  · 证书管理           │
│  · 签名预览          │           │  · CRL查询            │
│  · 验真二维码         │           │  · 审计记录           │
└────────────────────┘           └─────────┬───────────┘
                                           │
                                    ┌──────┴──────┐
                                    │  Key Store   │
                                    │  DB / HSM    │
                                    └─────────────┘
```

**关键依赖:**
- `gm-crypto` (Node.js 国密库) 或 `sm-crypto` (纯 JS)
- 或使用 Web Crypto API + 国密 polyfill 实现浏览器端 SM2/SM3
- 推荐: 服务端签名 + 前端展示的方案 (保证私钥安全)

## P0-02 COC 监管链

### 对标基准
StarLIMS/LabWare: 完整的 Chain of Custody 功能，从样品采集→运输→接收→存储→分样→检测→处置全程追踪。每一步记录操作人、时间、操作内容、交接签名。

### HC-LIMS 当前状态
❌ 未实现 — 样品登记有基础步骤，但无 COC 链式追踪。

### 功能说明

**核心功能点:**
1. **COC 表单生成**:
   - 样品登记时自动生成 COC 编号 (规则可配: 日期+流水号)
   - COC 表单包含: 样品信息、委托人、检测项目、采样地点/时间
   - 支持打印 COC 表单 (交接凭证)
2. **样品交接记录**:
   - 采样→送样→收样 每一步扫码交接
   - 记录: 交接双方签名 + 时间 + 样品状态
   - 支持委托方电子签名确认
3. **流转节点追踪**:
   - 登记 → 接收 → 分样 → 检测 → 留样 → 处置
   - 每个节点记录: 操作人、时间、操作内容、样品状态快照
   - 状态异常时自动告警 (长时间未流转)
4. **COC 完整性校验**:
   - 校验链式完整性: 每个节点必须有前序节点
   - 检测到跳转/缺失时标记异常
   - 支持 COC 审计报告导出
5. **样品处置记录**:
   - 留样到期自动提醒
   - 处置方式选择 (销毁/退回/续期)
   - 处置确认签名
6. **COC 可视化追踪**:
   - 时间线视图展示样品全生命周期
   - 高亮异常/延迟节点
   - 支持点击节点查看详细操作记录

### 技术方案

```
┌─────────────────────────────────────────────────────┐
│                  COC 管理器                          │
├──────────┬──────────┬──────────┬──────────┬─────────┤
│ COC生成  │ 交接管理  │ 节点追踪  │ 处置记录 │ 可视化  │
└──────────┴──────────┴──────────┴──────────┴─────────┘
       │          │          │          │          │
       └──────────┴──────────┴──────────┴──────────┘
                         │
                    ┌────┴────┐
                    │  Event  │
                    │  Store  │  ← 事件溯源 (Event Sourcing)
                    └─────────┘
```

**核心设计: 事件溯源 (Event Sourcing)**
- 每个 COC 节点是一个不可变事件
- 事件按时间顺序排列形成完整 COC 链
- 当前样品状态由事件序列聚合得出
- 杜绝篡改和跳转

### 数据结构

```typescript
// COC 事件 (每个节点一个事件)
interface COCEvent {
  id: string;
  sampleId: string;
  cocChainId: string;
  eventType: COCEventType;
  operatorId: string;
  operatorName: string;
  timestamp: string;
  location: string;
  signature?: string;        // 交接签名
  witnessSignature?: string; // 见证人签名
  notes?: string;
  metadata: Record<string, unknown>;  // 节点特有数据
  previousEventId: string;   // 前序事件 (链式校验)
}

type COCEventType = 
  | 'sampling'       // 采样
  | 'submission'     // 送样
  | 'receipt'        // 收样
  | 'registration'   // 登记
  | 'subsampling'    // 分样
  | 'testing'        // 检测
  | 'retention'      // 留样
  | 'disposal'       // 处置
  | 'return'         // 退回
  | 'exception';     // 异常

// COC 链摘要
interface COCChain {
  id: string;
  sampleId: string;
  cocNumber: string;
  status: COCStatus;
  events: COCEvent[];
  chainIntegrity: boolean;  // 链完整性校验
  integrityCheckAt: string;
}

type COCStatus = 'active' | 'completed' | 'broken' | 'disposed';
```

## P0-03 自定义字段/动态表单

### 对标基准
LabWare: Configurable Fields — 字段类型、校验规则、可见性、必填性均可通过配置界面管理，无需编码。
StarLIMS: 动态表单配置 — 支持表单模板、字段分组、条件显示。

### HC-LIMS 当前状态
❌ 硬编码字段 — 所有表单字段在 TypeScript 代码中硬编码，扩展字段需修改代码。

### 功能说明

**核心功能点:**
1. **JSON Schema 动态渲染**:
   - 所有表单基于 JSON Schema 动态渲染，而非硬编码 JSX
   - 支持字段类型: 文本、数字、日期、下拉选择、多选、文件上传、富文本、签名
   - 支持字段校验: 必填、格式、范围、正则
2. **字段配置管理界面**:
   - 按模块配置字段 (样品/检测/报告/仪器等)
   - 每个模块支持: 基础字段 + 自定义扩展字段
   - 字段属性: 名称、类型、必填、默认值、排序、分组、条件显示
3. **条件显示逻辑**:
   - 支持 `showIf: { field: "sampleType", value: "water" }`
   - 支持多条件组合: AND/OR 逻辑
   - 支持联动: 下拉 A 的值影响 下拉 B 的选项
4. **字段分组/布局**:
   - 字段可分组为: 基本信息、检测信息、附加信息
   - 支持 Tabs / Collapse / Step 布局
   - 每组的可见性、顺序可配
5. **自定义字段数据存储**:
   - 使用 JSONB/JSON 列存储扩展字段值
   - 支持索引查询
   - 支持导出 (自定义字段一并导出)
6. **字段模板**:
   - 按样品类型/检测类型预定义字段模板
   - 登记时选择模板自动加载字段配置
7. **版本管理**:
   - 字段变更记录版本历史
   - 已提交的样品使用创建时的字段快照

### 技术方案

```
前端                                后端
┌────────────────────────┐        ┌───────────────────┐
│  DynamicFieldRenderer  │        │  FieldConfig API  │
│  ┌──────────────────┐  │ REST  │  · CRUD字段配置    │
│  │ JSON Schema      │  │◄─────►│  · 模板管理        │
│  │ → RJSF / AntD    │  │       │  · 版本快照        │
│  │ → 动态渲染       │  │       └────────┬──────────┘
│  └──────────────────┘  │               │
│                        │        ┌───────┴───────┐
│  FieldConfigEditor     │        │  PostgreSQL   │
│  · 拖拽排序            │        │  JSONB字段     │
│  · 属性面板            │        └───────────────┘
│  · 预览模式            │
└────────────────────────┘
```

- `@rjsf/core` + `@rjsf/antd` — React JSON Schema Form，基于 JSON Schema 自动渲染表单

## P1-01 条码批量打印

### 对标基准
StarLIMS: Zebra/Sato 原生支持，模板定制，批量打印。
LabWare: 多打印机支持，标签模板管理。

### HC-LIMS 当前状态
⚠️ 有基础打印按钮，但仅单个打印，无模板、无批量、无打印机配置。

### 功能说明

**核心功能点:**
1. **条码/二维码生成**:
   - 支持 Code128、Code39、EAN-13 (条码)
   - 支持 QR Code (二维码)
   - 可编码: 样品编号、位置编码、试剂批号
2. **标签模板管理**:
   - 模板设计: 尺寸、字体、条码位置、文字内容
   - 预设模板: 样品标签、试剂标签、仪器标签、位置标签
   - 支持自定义文本字段 + 动态数据绑定
3. **批量打印**:
   - 选择多个样品/试剂 → 一键打印
   - 支持 Zebra ZPL 指令 / 通用 ESC/POS / PDF 生成
   - 打印机配置: IP/串口/USB, 纸张大小, 打印份数
4. **打印历史**:
   - 记录每次打印: 时间、内容、操作人、打印机
   - 支持补打 (保留最后一次打印记录)

### 技术方案

- 前端: `jsbarcode` + `qrcode.react` 生成条码
- 打印: 使用 `BrowserPrint` (Zebra) 或生成 PDF 调用系统打印
- 模板: 自研轻量标签模板渲染器
- 后端: 打印机配置存储 + 打印日志

| 天 | 任务 | 产出 |

### 数据结构

```typescript
// 流程定义
interface WorkflowDefinition {
  id: string;
  name: string;
  module: ModuleType;
  version: number;
  nodes: WorkflowNode[];
  edges: WorkflowEdge[];
  published: boolean;
}

interface WorkflowNode {
  id: string;
  type: 'start' | 'approval' | 'countersign' | 'notification' 
       | 'condition' | 'parallel' | 'timer' | 'end';
  label: string;
  config: {
    assignee?: string | string[];       // 指定人
    assigneeRole?: string;               // 指定角色
    assigneeExpr?: string;               // 动态表达式
    deadline?: number;                   // 超时(小时)
    escalationAction?: 'notify' | 'transfer' | 'autoApprove';
    rejectTargetId?: string;             // 驳回跳转
    conditionExpr?: string;              // 条件表达式
  };
}

// 流程实例
interface WorkflowInstance {
  id: string;
  definitionId: string;
  definitionVersion: number;
  businessType: string;      // report_approval | purchase | test
  businessId: string;        // 关联业务 ID
  status: 'running' | 'completed' | 'rejected' | 'terminated';
  currentNodeIds: string[];  // 当前节点
  tasks: WorkflowTask[];
  createdAt: string;
  completedAt?: string;
}

// 流程任务
interface WorkflowTask {
  id: string;
  instanceId: string;
  nodeId: string;
  assignee: string;
  status: 'pending' | 'approved' | 'rejected' | 'transferred';
  comment?: string;
  createdAt: string;
  actedAt?: string;
}
```
