## P0-1 委托管理 (Order Management)

### 🔴 系统专家意见

**1. 数据模型缺少关键字段**

| 缺失字段 | 影响 | 建议 |
|---------|------|------|
| `customerPoNo` (客户采购单号) | 客户对账时无法关联 | 必须添加 |
| `samplingAddress` (采样地址) | 环境监测/现场采样场景必需 | 必须添加 |
| `isRush` (是否加急) | 与 `urgency` 不同——加急意味着额外收费 | 建议添加 |
| `billingStatus` (计费状态) | 财务对账需要 | 建议添加 |
| `submittedAt` / `receivedAt` | 时间线追踪 | 必须添加 |
| `assignedLabId` (指定实验室) | 客户可能指定特定实验室 | 建议添加 |

**2. API 幂等性缺失**

`POST /api/v1/orders` 没有幂等键。网络重试可能导致重复创建委托。建议：
```
POST /api/v1/orders
Header: Idempotency-Key: <uuid>
```

**3. 状态机过于简化**

当前状态: `draft → submitted → received → in_progress → review → completed → archived`

缺失状态：
- `cancelled`：客户取消委托
- `rejected`：技术审核拒绝
- `on_hold`：客户暂停/样品问题暂停
- `partial_complete`：部分样品完成

建议完整状态机：
```
draft → submitted → tech_review → accepted → received → in_progress
                                                    ↘ rejected (退回修改)
                                     ↘ on_hold (暂停)
in_progress → partial_complete → completed → archived
                            ↘ cancelled
```

**4. 变更记录缺少 diff 精度**

`beforeValue: string` 无法精确表达复杂变更（如"追加了2个检测项目，删除了1个"）。建议：
```typescript
interface OrderChange {
  changeType: string;
  patches: JSONPatch[];  // RFC 6902 JSON Patch
}
```

**5. 缺少委托模板**

重复委托（如每月水质监测）应支持从模板创建，减少录入工作。金现代和 SampleManager 都支持此能力但设计文档未包含。

---

### 🟡 资深用户意见

**1. "我每天要接 20 个委托电话"**

> 业务员最常用的不是表单，而是**快速委托**模式：
> - 选择老客户 → 自动带出历史项目和检测套餐
> - 输入样品数量 → 一键生成委托
> - 整个过程 < 30 秒

设计文档的"创建委托 Modal"对高频操作太慢。建议增加**快速委托**入口。

**2. "客户经常改主意"**

> 真实场景：客户电话说"再加一个铅的检测"。当前设计需要打开委托 → 编辑 → 添加项目 → 保存。

建议在委托列表直接支持**行内快捷追加**，或右键菜单"追加检测项目"。

**3. "我需要知道哪些委托快超期了"**

> SLA 倒计时：应在列表上用颜色标记（绿色=正常，黄色=<2天，红色=已超期）

**4. 缺少检测套餐/方案推荐**

> 环境监测站最常见的场景：客户说"我要做地表水检测"，业务员需要手动选择 20+ 个项目。应该有**预设检测方案**（如"地表水24项"、"饮用水106项"）一键选择。

## P0-2 ELN 模板引擎

### 🔴 系统专家意见

**1. 公式引擎设计过度简化**

设计文档中 `formula: "(absorbance - blank) * K * dilution"` 是字符串解析方案，存在严重问题：
- 无法处理复杂公式（对数、指数、条件分支）
- 变量名冲突风险
- 无法做编译期校验

**建议方案**：
```typescript
interface FormulaNode {
  type: 'binary' | 'unary' | 'literal' | 'variable' | 'function';
  operator?: '+' | '-' | '*' | '/' | '^' | 'log' | 'ln' | 'sqrt';
  left?: FormulaNode;
  right?: FormulaNode;
  value?: number;
  varName?: string;
}
```
采用 AST 树而非字符串解析，或至少使用 `math.js` 安全沙箱。

**2. 修约规则需要符合 GB/T 8170**

检测行业的修约不是简单的 `toFixed(n)`。需要实现：
- 四舍六入五成双（GB/T 8170-2008）
- 有效数字 vs 小数位数
- 单位换算后修约

```typescript
interface RoundingRule {
  method: 'gb8170' | 'significant_figures' | 'decimal_places';
  value: number;
}
```

**3. 模板版本冻结机制缺失**

> "模板升级后，历史记录不应变化" ——但当前设计没有说明如何实现。

建议：
- 模板发布时生成不可变快照（`templateVersion`）
- ELN 记录绑定 `templateId + templateVersion`
- 新记录自动使用最新发布版本

**4. 缺少模板导入/导出**

> 多实验室部署时，理化实验室创建的 COD 模板，环境实验室也需要用。必须有模板导入/导出（JSON 格式）。

**5. 字段联动逻辑不完整**

设计文档提到字段联动，但没有说明联动规则引擎。建议：
```typescript
interface FieldDependency {
  targetFieldId: string;
  condition: {
    sourceFieldId: string;
    operator: 'eq' | 'neq' | 'gt' | 'lt' | 'in' | 'contains';
    value: any;
  };
  action: 'show' | 'hide' | 'setValue' | 'setOptions';
  actionValue?: any;
}
```

---

### 🟡 资深用户意见

**1. "我不需要拖拽，我需要复制上次的"**

> 对于实验员来说，90% 的实验是重复性的。最重要的功能不是拖拽设计器，而是**一键加载上次模板并自动填入上次的参数**。

建议增加"从历史复制"功能——打开任务 → 选择"复用上次记录" → 自动填入。

**2. "公式我看不懂，但结果对不对我知道"**

> 实验员不关心中间公式，他们关心的是：
> - 这个结果合理吗？（历史范围对比）
> - 需要重做吗？（自动提示）

建议在每个计算结果旁显示：**历史正常范围**（灰色）和**本次结果**（蓝色），异常时红色高亮。

**3. "SOP 步骤太多，能不能自动跳过已完成的"**

> 如果 SOP 有 8 步，但步骤 3 是可选的（如"如样品澄清则跳过过滤"），系统应自动判断并跳过。

## P0-3 自动派工规则引擎

### 🔴 系统专家意见

**1. 缺少设备能力矩阵**

当前设计以"人"为核心派工。但现实中 80% 的检测是**仪器驱动**的：
- HPLC 只能做液相项目
- 一台 ICP-MS 有 4 个通道
- 设备校准到期不可用

建议数据模型增加 `InstrumentCapability`：
```typescript
interface InstrumentCapability {
  instrumentId: string;
  supportedMethods: string[];
  supportedTestItems: string[];
  channelCount: number;
  calibrationStatus: 'valid' | 'expiring' | 'expired';
  maintenanceStatus: 'available' | 'scheduled' | 'in_maintenance';
}
```

**2. 排程算法需要明确定义**

"负载均衡"太模糊。需要具体算法：

```
Priority Score = w1 * urgencyWeight + w2 * (1 / remainingTAT) + w3 * (1 / analystLoad)
```

其中 `w1=0.5, w2=0.3, w3=0.2`（可通过规则配置调整）。

**3. 缺少重排程触发条件**

什么情况下触发重排程？
- 设备故障
- 检测员请假
- 紧急插单
- 前序任务超时

这些触发条件必须在设计中明确。

**4. 并发安全问题**

2 个样品同时到达 → 2 个派工请求 → 同时查询负载 → 都分配给同一人 → 瞬间超载。

建议采用**悲观锁**或**队列串行化**。

---

### 🟡 资深用户意见

**1. "自动派工很好，但我需要能手动调整"**

> 自动派工 + 人工微调 = 最佳方案。派工结果应显示在甘特图/看板上，主管可以拖拽调整。

**2. "周五下午 4 点收到 20 个急样怎么办"**

> 真实场景痛点。系统应提供：
> - 一键"全部紧急"批量提升
> - 预估完成时间（考虑周末和当前负载）
> - 建议：哪些检测可以推迟（低优先级、TAT 尚早）

**3. "不同的人做同样的项目速度不一样"**

> 派工规则应考虑**历史 TAT**：
> - 张伟做 COD 平均 2.3 小时
> - 李明做 COD 平均 3.1 小时
> - 紧急任务优先分配给张伟

## P0-4 质控方案配置

### 🔴 系统专家意见

**1. Westgard 规则实现有逻辑漏洞**

`2_2s` 和 `R_4s` 规则设计文档中的实现只检查"连续 2 个点"，但 Westgard 规则要求的是"**同一批次内**连续 2 个质控值"，而非跨批次。需要区分：

```typescript
// 错误实现：跨批次检查
if (Math.abs(values[i] - mean) > 2*sd && Math.abs(values[i-1] - mean) > 2*sd)

// 正确实现：仅同批次内检查
const batchValues = values.filter(v => v.batchId === currentBatchId);
```

**2. 缺少质控样品独立台账**

质控样品（CRM/标准物质）本身需要管理：
- 批次号、浓度、不确定度
- 有效期、开封效期
- 证书编号、供应商
- 存储条件

当前设计把质控样当作配置项，但实际上质控样是独立实体。

**3. QC 频率计算边界条件**

"每 10 个样品插入 1 个质控"——边界情况：
- 只有 3 个样品时是否插入？
- 批次开始时强制插入空白 → 算不算在频率计数内？
- 加标回收和平行样是"每批次"还是"每 N 个"？

**4. 缺少 QC 结果趋势预警**

Westgard 规则是"即时判异"，但更重要的是**趋势预判**：
- 连续 7 个点在同一侧（即使未超限）→ 系统偏差
- 连续 6 个点递增或递减 → 漂移预警

JXD 的"分钟级判异 + 趋势预警"应该包含此能力。

---

### 🟡 资深用户意见

**1. "空白超了怎么办，我重跑一次？"**

> 真实场景：空白值偏高 → 实验员重跑空白 → 第二针正常 → 用第二针？用第一针？
> 
> 系统应明确规则：质控失败后允许重跑，但**所有质控结果必须保留**，不能选择性删除。

**2. "月初配的质控样，月底还能用吗"**

> 质控样开封效期管理缺失。系统应在实验员使用质控样时检查效期。

**3. "评审专家要看我们过去 3 年的质控图"**

> 质控数据需按年度/季度归档，支持历史查询和导出。L-J 图应支持选择任意时间范围。

## P0-5 RBAC 权限矩阵

### 🔴 系统专家意见

**1. 缺少权限冲突检测**

当一个用户有多个角色时:
- 角色A：可编辑报告
- 角色B：报告只读
- 用户同时有 A 和 B → 能否编辑？

应定义权限合并策略：**allow-override**（允许优先）或 **deny-override**（拒绝优先）。

建议采用 **deny-override**（安全优先原则）。

**2. 数据权限过滤缺少性能考虑**

"数据范围=本实验室" → SQL 层添加 `WHERE lab_id = ?`。但如果用户跨实验室查询（如 QA 看全部），需要在 API 层动态处理，而非 DB 层。

建议采用**数据权限中间件**模式：
```
请求 → 认证 → 权限中间件(注入 scope) → 业务层(自动过滤) → 响应
```

**3. 缺少临时权限委托**

"主管休假 2 周，需要委托审批权给副主管"。

建议：
```typescript
interface PermissionDelegation {
  fromUserId: string;
  toUserId: string;
  permissions: string[];
  validFrom: string;
  validUntil: string;
  reason: string;
}
```

**4. API 权限粒度不够**

当前 API 设计只有角色 CRUD。缺少：
- `GET /api/v1/auth/check-permission?action=edit&resource=report&id=123`
- `GET /api/v1/users/:id/effective-permissions`（合并多角色后的最终权限）

**5. 缺少密码策略配置**

21 CFR Part 11 要求：
- 密码复杂度（大小写+数字+特殊字符）
- 90 天过期
- 5 次失败锁定
- 历史密码不可重复

---

### 🟡 资深用户意见

**1. "同一个实验室的人互相能看结果吗"**

> 这是最常见的权限问题。需要明确：
> - 同实验室检测员能否看到彼此的结果？→ 建议可以
> - 不同实验室检测员能否看到？→ 建议不可以
> - QA 能否看到所有？→ 可以

**2. "我需要知道谁动过我的权限"**

> 每次权限变更应有通知：邮件/站内信告知被变更用户。

**3. "新人来了，我要给他配 5 个角色太麻烦了"**

> 建议支持**角色模板**（如"新入职检测员 = 检测员 + 设备使用者 + 样品查看者"）。
